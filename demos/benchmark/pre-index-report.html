What it means for the path

  • →
    Governance is free at the compute layer.

    Sub-millisecond per call means deterministic control sits inline on the data path with no latency-budget conversation. The engine keeps up with the wire.

  • →
    Your latency is a networking problem, not a compute one.

    ~97% of the observed per-call time is TLS setup and round-trip. Shorter, warmer, more direct paths translate almost 1:1 into end-to-end improvement.

  • →
    The first win is connection reuse.

    Pooling connections alone is a 3× per-call improvement here, before touching anything else.

  • →
    Two engines, one policy, same answer.

    Identical output across Themis and Aergia means the control is deterministic and portable — the choice between engines is about placement and throughput, not behavior.

Method. 1,000-chunk synthetic pre-index corpus. "Do nothing" = forward everything; "RE2" = the kit's local Go regex baseline (pattern masking + boilerplate strip); "NOL8" = the same literal known-value policy deployed to Themis (:443) and Aergia (:444), measured over the network via the adapter. Governance counts are exact ("chunks touched" for RE2 = masked + dropped; "chunks governed" for NOL8 = chunks containing a governed value). RE2 and NOL8 target different content, so their token counts are not a like-for-like comparison. Latency decomposed from the field client box, N = 100 per layer, medians; engine time is bounded as (warm − network RTT) and lands within network jitter, so it is an upper bound — no server-side timing hook is exposed. Cold = new TLS connection per request (current adapter); warm = pooled. Not a concurrency/throughput saturation test.
